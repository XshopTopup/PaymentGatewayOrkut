Dokumentasi API Pembayaran QRIS

API yang dioptimalkan untuk transaksi pembayaran berbasis QRIS

Pendahuluan

API Pembayaran QRIS memungkinkan pembuatan dan verifikasi transaksi pembayaran berbasis QRIS. API ini menjamin keunikan Transaction ID dan nominal dalam periode 24 jam untuk mencegah konflik. Fitur optimasi meliputi per-user request delay untuk mencegah pemrosesan simultan, caching untuk pengecekan status pembayaran, pengurangan pengambilan data API, dan pembersihan otomatis data yang kadaluarsa. Dokumentasi ini merinci endpoint, parameter, respons, dan fitur optimasi.

Endpoint API

POST /api/topup

Menghasilkan kode QRIS untuk transaksi pembayaran dengan nominal dan Transaction ID yang unik. Transaction ID diformat sebagai YYYYMMDD (contoh: 202505170001), memastikan keunikan dalam periode 24 jam. Sufiks unik (1-500) ditambahkan ke nominal untuk membuat total unik. Setiap permintaan dari pengguna berbeda memiliki jeda minimal 10 detik untuk mencegah konflik. Maksimal 1 transaksi aktif per pengguna dan 5 transaksi aktif secara global.

Body Permintaan

{
    "amount": number, // Wajib, bilangan bulat positif (contoh: 10000)
    "username": string // Wajib, pengenal pengguna (contoh: "john_doe")
}

Respons

{
    "id": string, // Transaction ID unik (contoh: "202505170001")
    "qris": string, // URL gambar QRIS (contoh: "https://payorkut.ceknickname.my.id/qris/XST1234567890.jpg")
    "expireAt": number, // Timestamp kadaluarsa (milidetik)
    "expiresIn": number, // Sisa waktu kadaluarsa (detik)
    "amount": number, // Nominal asli yang diminta
    "kodeunik": number, // Sufiks unik (1-500)
    "total": number, // Total nominal yang harus dibayar (amount + kodeunik)
    "note": string, // Instruksi pembayaran
    "expiryStatus": string // "active" atau "expires_soon" (jika <10 menit)
}

Respons Error

  • 400 Bad Request: Nominal tidak valid atau bukan bilangan bulat positif, atau username tidak diberikan.
  • 429 Too Many Requests:
    • Terlalu banyak transaksi aktif (1 per pengguna, 5 secara global).
    • Tidak ada nominal unik yang tersedia.
    • Permintaan tertunda karena jeda 10 detik antar pengguna.
  • 500 Internal Server Error: Gagal menghasilkan QRIS.

Contoh

curl -X POST https://payorkut.ceknickname.my.id/api/topup \
-H "Content-Type: application/json" \
-d '{"amount": 10000, "username": "john_doe"}'

GET /api/check-payment/:id

Memeriksa status transaksi pembayaran berdasarkan Transaction ID. Menggunakan respons API yang di-cache (validitas 30 detik) dan hanya mengambil 10 transaksi terbaru untuk mengurangi latensi dan beban server.

Parameter

  • id: Transaction ID (contoh: "202505170001")

Respons

{
    "status": string, // "paid" (dibayar), "pending" (menunggu), atau "expired" (kadaluarsa)
    "expiresIn": number, // Sisa waktu kadaluarsa (detik, 0 jika paid atau expired)
    "expiryStatus": string, // "active" atau "expires_soon" (jika <10 menit, hanya untuk pending)
    "paymentDetails": { // Hanya disertakan jika status "paid"
        "paymentId": string, // Referensi pembayaran dari gateway
        "paymentTime": string, // Timestamp pembayaran
        "paymentMethod": string, // Metode pembayaran (contoh: merek QRIS)
        "amount": number, // Nominal asli
        "uniqueAmount": number, // Total nominal yang dibayar
        "uniqueSuffix": number // Sufiks unik
    }
}

Respons Error

  • 404 Not Found: Transaksi tidak ditemukan.
  • 500 Internal Server Error: Gagal memeriksa status pembayaran.

Contoh

curl https://payorkut.ceknickname.my.id/api/check-payment/202505170001

GET /qris/:filename

Mengambil gambar QRIS untuk transaksi.

Parameter

  • filename: Nama file gambar QRIS (contoh: "XST1234567890.jpg")

Respons

Mengembalikan file gambar QRIS (JPEG).

Respons Error

  • 404 Not Found: File tidak ditemukan.

Optimasi API

API ini mencakup beberapa optimasi untuk memastikan performa, keandalan, dan skalabilitas:

  • Per-User Request Delay: Permintaan deposit dari pengguna berbeda diproses dengan jeda minimal 10 detik untuk mencegah konflik dalam pembuatan nominal unik dan mengurangi beban pada layanan QRIS eksternal.
  • Transaction ID Unik: Transaction ID dibuat dalam format YYYYMMDD (contoh: 202505170001), di mana YYYYMMDD adalah tanggal dan counter adalah nomor urut harian (4 digit, hingga 9999). Ini menjamin tidak ada duplikasi ID dalam 24 jam.
  • Nominal Unik: Sufiks unik (1-500) ditambahkan ke nominal yang diminta untuk membuat total unik (contoh: 10000 + 123 = 10123). Ini mencegah konflik dalam periode 24 jam. Jika tidak ada sufiks unik yang tersedia setelah 999 percobaan, API mengembalikan error 429.
  • Batas Transaksi: Maksimal 1 transaksi aktif per pengguna dan 5 transaksi aktif secara global untuk mencegah overload sistem. Jika batas tercapai, API mengembalikan error 429 dengan waktu tunggu.
  • Pengecekan Status yang Dioptimalkan: Endpoint /api/check-payment/:id hanya mengambil 10 transaksi terbaru dari API eksternal untuk mengurangi latensi dan beban server. Respons API di-cache selama 30 detik untuk meminimalkan panggilan berulang.
  • Pembersihan Otomatis: Transaksi yang kadaluarsa, nominal unik, counter ID, dan waktu permintaan pengguna dibersihkan setiap 2 menit. Catatan transaksi yang telah diproses dihapus setiap jam jika melebihi 1000 entri. Gambar QRIS dihapus setelah pembayaran atau kadaluarsa untuk menghemat penyimpanan.

Catatan Tambahan

  • Transaksi kadaluarsa setelah 10 menit.
  • Transaction ID dan nominal dijamin unik dalam periode 24 jam.
  • Gambar QRIS dihapus otomatis setelah pembayaran atau kadaluarsa.
  • Permintaan deposit dari pengguna berbeda memiliki jeda minimal 10 detik.
  • Untuk volume transaksi tinggi, pertimbangkan menggunakan database (contoh: Redis) untuk menyimpan nominal unik, counter ID, dan waktu permintaan pengguna.
  • Kunci API dan Merchant ID harus disimpan dengan aman dalam variabel lingkungan.
